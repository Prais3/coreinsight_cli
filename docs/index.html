Vision

The beginning of
self-improving software

Today, CoreInsight profiles your code and verifies AI-generated optimizations against hardware reality. But the direction is bigger than that.

Every verified optimization is stored and recalled. Every retry loop teaches the tool what works on your hardware. Every benchmark is a data point in a growing body of mathematical evidence about how code should run on silicon.

"The goal is software that understands its own performance well enough to improve itself — autonomously, verifiably, and without needing a human in the loop."

That is a hard problem. CoreInsight is the first step: an AI that does not just generate code, but generates code it can prove is correct and faster. Not because the LLM said so. Because the hardware said so.

Architecture

Three pillars that
make it trustworthy

01 — VERIFICATION

Two-pillar sandbox

Every optimization runs through independent checks inside Docker: speedup integrity (recomputed from raw timing) and output correctness (identical test inputs, float tolerance). Hallucinated speedups are flagged and rejected.

02 — EVIDENCE

AI-free hardware profiling

On top of sandbox verification, real tools — cProfile, perf stat — measure wall time, function calls, cache misses, and CPU cycles. These numbers come from silicon, not a language model. Deterministic. Auditable. Pro feature.

03 — MEMORY

Optimization memory

A local vector database stores every verified result. On new analyses, semantically similar past optimizations are retrieved before the LLM is called. A match skips the entire pipeline and returns instantly — with full provenance.

Pipeline

What happens when
you run an analysis

01

Memory lookup new

SHA-256 hash and semantic similarity search against every previously verified optimization. On a hit, skip steps 02–05 entirely.

02

AST parsing + RAG context

tree-sitter extracts functions. ChromaDB retrieves cross-file dependencies so the LLM understands your structs, types, and helpers.

03

Hardware-aware LLM analysis

Detected CPU/GPU specs are passed to the model alongside your code. The LLM identifies bottlenecks and writes an optimized replacement.

04

Autonomous retry loop

Generated harness runs in Docker. If it fails or produces no real speedup, the LLM is sent the error and asked to fix it — up to 5 times.

05

Mathematical verification

Speedup is recomputed from raw timing columns and cross-checked. Correctness is tested against LLM-generated inputs in a fresh container.

06

AI-free hardware evidence pro

cProfile and perf stat run against both versions. Hardware counters are written to the report as deterministic, LLM-independent proof.

07

Store to memory

The verified result, speedup, and profiler evidence are stored in the local vector database. The next time a similar function is seen, the pipeline starts at step 01.

v0.2.0

The tool gets smarter
every time you use it

Optimization memory is the feature that moves CoreInsight from a stateless code tool toward something that accumulates expertise. Every verified run contributes to a local knowledge base that grows with your codebase.

Local optimization store — ~/.coreinsight/memory_db

⚡ Instant recall on exact and near-exact function matches — no LLM call, no Docker spin-up
100% local — the store lives on your machine, never transmitted anywhere
SHA-256 exact hash match for identical functions, semantic cosine similarity for structural variants
Full provenance every recalled result carries its original speedup, timestamp, and hardware evidence

Inspect the store: coreinsight memory  ·  Clear it: coreinsight memory --clear
